# 思路与直觉

> 对应代码：`data_generation/clustering.py`、`pipelines/clustering/dbscan.py`
>
> 对比对象：`docs/clustering/kmeans/`

## 本章目标

1. 用直观方式理解 DBSCAN 到底在做什么。
2. 理解为什么它在当前双月牙数据上效果更好。
3. 理解它与 KMeans 在聚类思路上的关键差异。

## 重点方法与概念速览

| 名称 | 类型 | 作用 |
|---|---|---|
| 密度聚类 | 任务 | 根据局部样本密度与连通关系发现簇 |
| 核心点 | 几何直觉 | 周围邻居足够多、可以向外扩展簇的点 |
| 噪声点 | 输出结果 | 不属于任何高密度区域的样本 |
| 双月牙数据 | 示例数据 | 当前仓库里最适合演示 DBSCAN 的非线性簇数据 |
| KMeans | 对比算法 | 更擅长球形簇和中心式切分 |

## 1. 为什么需要 DBSCAN

当数据的簇不是圆团状，而是弯曲、拉长或形状不规则时，用“距离最近中心”来切分数据往往并不自然。

DBSCAN 给出的思路是：

- 不先找中心。
- 先找局部上足够密集的点。
- 再沿着密度连通关系把整片区域扩展开来。

### 理解重点

- 这是一种“从局部密度出发”的聚类方法，而不是“从全局中心出发”的方法。
- 因为它不依赖固定形状的中心假设，所以更适合处理任意形状簇。
- 同时，它还能自然地识别孤立点和异常点。

## 2. 为什么当前仓库示例里它表现很好

当前 DBSCAN 数据来自：

```python
make_moons(
    n_samples=400,
    noise=0.08,
    random_state=42,
)
```

这类数据的特点是：

- 簇形状弯曲
- 线性不可分
- 不适合用单一中心代表
- 维度只有 2，便于直观看图

### 理解重点

- 这类数据正是 DBSCAN 的典型优势场景。
- 如果用 KMeans，它更可能按中心距离把两个月牙错误切开。
- DBSCAN 则可以沿着月牙形状中的高密度区域，把每个弯月完整扩展成一个簇。

## 3. 用“从核心点向外生长”理解算法

可以把 DBSCAN 理解成一个扩展过程：

1. 找到一个核心点。
2. 把它邻域内能连上的点都吸收进来。
3. 如果新加入的点里还有核心点，就继续向外扩展。
4. 直到这片高密度区域不能再扩展为止。

### 示例代码

```text
遍历样本点
  如果当前点是核心点:
    创建新簇
    沿密度可达关系不断扩展
  否则暂不扩展
最终无法归入任何簇的点标为噪声
```

### 理解重点

- DBSCAN 更像是在“长出一片簇”，而不是“先放几个中心再把点分过去”。
- 所以它特别适合处理曲线状、连通状或局部形状复杂的数据。

## 4. 为什么标准化会影响结果

在当前流水线中，训练前有这样一步：

```python
scaler = StandardScaler()
X_scaled = scaler.fit_transform(X)
```

### 理解重点

- `eps` 本质上是距离阈值，因此对尺度非常敏感。
- 如果不同特征量纲差异过大，同一个 `eps` 在不同轴上的含义会不一致。
- 所以即便当前示例数据较规整，文档仍要明确强调标准化的重要性。

## 5. 与 KMeans 的直觉差异

两者的核心直觉差异可以这样理解：

| 算法 | 更擅长的数据形状 | 核心依据 |
|---|---|---|
| KMeans | 球形、凸形、方差接近的簇 | 到中心的距离 |
| DBSCAN | 任意形状、含噪声点的数据 | 局部密度与连通性 |

### 理解重点

- KMeans 看到的是“哪个中心离你更近”。
- DBSCAN 看到的是“你周围是否足够密、能否和附近区域连成一片”。
- 因此 `make_blobs` 更适合 KMeans，而 `make_moons` 更能体现 DBSCAN 的优势。

## 常见坑

1. 因为当前示例效果很好，就误以为 DBSCAN 适合所有聚类数据。
2. 把噪声点理解成算法错误，而不是算法主动识别出的离群样本。
3. 只调 `eps`，却忽略 `min_samples` 也会显著影响簇结构。
4. 不区分“局部密度连通”与“全局几何距离接近”的差异。

## 小结

- DBSCAN 的直觉非常鲜明：从局部高密度区域出发，把能连成一片的样本扩展成簇。
- 当前仓库使用的双月牙数据与这种思路高度匹配，因此很适合做入门演示。
- 如果数据簇密度差异很大，或者参数选择不合适，DBSCAN 也可能表现不稳定。
