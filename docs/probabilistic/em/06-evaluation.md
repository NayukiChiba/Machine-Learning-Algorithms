# 评估与诊断

> 对应代码：`pipelines/probabilistic/em.py`、`result_visualization/cluster_plot.py`、`model_training/probabilistic/em.py`
>  
> 相关对象：`plot_clusters(...)`、`model.lower_bound_`

## 本章目标

1. 明确当前仓库实际使用了哪些评估手段，而不是泛泛讨论所有聚类指标。
2. 理解聚类分布图和训练日志分别能帮助我们诊断什么。
3. 明确当前实现没有做哪些聚类指标输出，以免误读源码能力边界。

## 重点方法与概念速览

| 名称 | 类型 | 作用 |
|---|---|---|
| `plot_clusters(...)` | 函数 | 生成预测标签与真实标签对比散点图 |
| `labels_pred` | 预测结果 | 当前 GMM 输出的簇标签 |
| `labels_true` | 对比标签 | 数据生成时记录的真实分量编号 |
| `model.lower_bound_` | 属性 | 训练收敛后的平均对数似然下界 |

## 1. 当前实现真正做了什么评估

### 参数速览（本节）

适用评估手段（本节）：

1. 聚类分布图
2. 训练日志

| 评估方式 | 来源 | 用途 |
|---|---|---|
| 聚类分布图 | `plot_clusters(...)` | 观察预测簇结构是否贴近真实分量分布 |
| 训练日志 | `train_model(...)` | 观察分量数、协方差结构和收敛下界 |

### 理解重点

- 当前 EM 流水线没有显式打印 `ARI`、`NMI`、`silhouette score`、`BIC`、`AIC` 等数值指标。
- 这并不表示这些指标不重要，而是说明本仓库当前实现更强调图像化对比与训练收敛日志。
- 因此阅读这一分册时，不能把“聚类指标表”想成已经在源码里实现的内容。

## 2. 聚类分布图是怎么生成的

### 参数速览（本节）

适用函数：`plot_clusters(X, labels_pred, labels_true=None, centers=None, feature_names=None, title='聚类分布', dataset_name='default', model_name='model', figsize=(12, 5))`

| 参数名 | 本例取值 | 说明 |
|---|---|---|
| `X` | `X_scaled` | 标准化后的二维特征 |
| `labels_pred` | `labels_pred` | 预测簇标签 |
| `labels_true` | `y_true` | 真实分量标签，用于对比 |
| `dataset_name` | `"em"` | 输出目录名 |
| `model_name` | `"gmm"` | 输出文件名前缀 |

### 示例代码

```python
plot_clusters(
    X_scaled,
    labels_pred=labels_pred,
    labels_true=y_true,
    title="EM (GMM) 聚类分布",
    dataset_name=DATASET,
    model_name=MODEL,
)
```

### 理解重点

- 当前图像会并排画出“预测标签”和“真实标签”两张散点图。
- 左图帮助你看模型实际聚出来的簇结构，右图帮助你对照数据生成时的真实分量分布。
- 对当前二维混合高斯数据来说，这种对比是最直观的评估方式。

## 3. 看聚类分布图时重点观察什么

### 参数速览（本节）

适用观察点（本节）：

1. 簇边界是否大致分开
2. 预测簇与真实分量是否大致对应
3. 是否出现明显混簇现象

| 现象 | 可能含义 |
|---|---|
| 预测簇形状和真实分量大致一致 | 当前 GMM 建模较合理 |
| 两个分量被合并成一个簇 | `n_components` 可能设得过少 |
| 一个真实分量被拆成多个簇 | `n_components` 可能设得过多 |
| 某片区域标签混杂明显 | 分量重叠较强或参数设定不合适 |

### 理解重点

- EM 的聚类标签编号本身不重要，重要的是空间分布是否大致匹配。
- 即使标签数字顺序不同，只要空间结构对得上，聚类结果仍可能是合理的。
- 因此看图时应优先看簇形状和分布，而不是逐个比较标签编号。

## 4. `lower_bound_` 在评估中应该怎么理解

### 参数速览（本节）

适用属性：`model.lower_bound_`

| 属性名 | 当前含义 |
|---|---|
| `lower_bound_` | 训练收敛后的平均对数似然下界 |

### 示例代码

```python
print(f"log-likelihood: {model.lower_bound_:.4f}")
```

### 理解重点

- 当前日志中的 `log-likelihood` 反映的是训练优化过程的下界水平。
- 它可以帮助判断模型是否收敛到一个相对稳定的解，但不能直接替代聚类质量可视化。
- 也就是说，`lower_bound_` 更偏“训练诊断”，而不是“最终聚类效果评分”。

## 5. 当前实现没有做什么

### 参数速览（本节）

当前源码未包含的内容：

1. 显式聚类指标打印
2. AIC / BIC 对比分量数
3. 责任度可视化
4. 学习曲线

| 未实现项 | 当前状态 |
|---|---|
| `ARI` / `NMI` / `silhouette score` 打印 | 未在流水线中出现 |
| `AIC` / `BIC` 选择分量数 | 未在流水线中出现 |
| `predict_proba(...)` 责任度图 | 未做教学展示 |
| 学习曲线 | 当前流程未调用相关函数 |

### 理解重点

- 评估章节必须以源码为准，不能把“EM 常见分析手段”写成“当前仓库已经实现”。
- 当前实现的评估重点是聚类对比图和训练日志，而不是数值指标面板。
- 如果后续扩展这部分，最自然的方向是补聚类指标打印、AIC/BIC 或责任度可视化。

## 常见坑

1. 只看 `lower_bound_`，不看聚类分布图，误把“似然更高”当成“图上一定更合理”。
2. 逐个比较标签编号，忽略聚类标签本身可交换这一事实。
3. 误以为当前流水线已经输出了 ARI、NMI、BIC 等完整指标，实际源码并没有这一步。

## 小结

- 当前 EM / GMM 的评估主线由两部分组成：聚类分布图和训练日志。
- 聚类分布图负责看空间结构是否合理，训练日志负责看模型设定和收敛下界。
- 只有把这两条线索一起看，才能更完整地理解当前实现的表现。
