# 思路与直觉

> 对应代码：`data_generation/probabilistic.py`、`model_training/probabilistic/hmm.py`
>  
> 相关对象：`ProbabilisticData.hmm()`、`train_model(...)`

## 本章目标

1. 理解 HMM 为什么把序列建模成“隐状态驱动观测”的过程。
2. 理解 HMM 与普通独立样本模型在建模思路上的根本区别。
3. 把这些直觉和当前仓库的离散观测序列数据对应起来。

## 重点方法与概念速览

| 名称 | 类型 | 作用 |
|---|---|---|
| 隐状态 | 概念 | 描述系统内部不可直接观测的状态 |
| 观测符号 | 概念 | 在每个时间步真正看到的输出 |
| 转移矩阵 | 参数 | 描述隐状态如何随时间演化 |
| 发射矩阵 | 参数 | 描述隐状态如何生成观测 |

## 1. HMM 想表达什么

HMM 的核心假设是：我们观察到的序列不是直接独立生成的，而是由一条看不见的隐状态链在背后驱动产生的。

### 参数速览（本节）

适用概念（本节）：

1. 隐状态序列
2. 观测序列
3. 时间依赖

| 概念 | 直观含义 |
|---|---|
| 隐状态 | 系统真实所处模式，但外部看不到 |
| 观测 | 当前时刻能直接记录到的符号 |
| 序列依赖 | 下一时刻的状态受上一时刻影响 |

### 理解重点

- 如果把每个时间点都当成独立样本，很多时序信息就会丢失。
- HMM 的价值，就是同时建模“状态如何转移”和“状态如何发射观测”。
- 当前分册里的 `obs` 和 `state_true` 正好对应这两层结构。

## 2. 为什么要区分“隐状态”和“观测”

### 参数速览（本节）

适用列（本节）：

1. `obs`
2. `state_true`

| 列名 | 当前含义 |
|---|---|
| `obs` | 训练时可见的离散观测符号 |
| `state_true` | 数据生成时记录的真实隐状态，仅用于训练后对比 |

### 理解重点

- 对模型来说，训练时真正能看到的只有 `obs`。
- `state_true` 虽然在数据文件里存在，但那是为了教学和对比效果，当前训练过程不会使用它。
- 也就是说，HMM 分册真正要解决的问题是“只看观测，去推断隐藏状态结构”。

## 3. 转移矩阵和发射矩阵的直观含义

### 参数速览（本节）

适用参数（本节）：

1. `hmm_A`
2. `hmm_B`

| 参数 | 直观含义 |
|---|---|
| 转移矩阵 `A` | 系统从一个隐状态走到另一个隐状态的概率 |
| 发射矩阵 `B` | 某个隐状态下更容易生成哪些观测符号 |

### 理解重点

- 转移矩阵决定“状态怎么变”。
- 发射矩阵决定“在某个状态下更容易看到什么”。
- 当前数据生成函数正是先按 `A` 采样状态，再按 `B` 采样观测，所以非常适合讲 HMM 的两层结构。

## 4. 为什么 HMM 比普通聚类或分类更适合序列

### 参数速览（本节）

适用对比对象（本节）：

1. 独立样本模型
2. HMM

| 维度 | 独立样本模型 | HMM |
|---|---|---|
| 样本关系 | 默认彼此独立 | 显式建模时间依赖 |
| 输入结构 | 一组无序样本 | 一条或多条有序序列 |
| 隐状态 | 通常不建模 | 核心组成部分 |
| 输出解释 | 单点预测 | 序列级状态路径与概率结构 |

### 理解重点

- HMM 的优势不在于“单个点预测得更准”，而在于能利用序列上下文。
- 当前分册里的 `lengths = [len(obs)]` 就在强调：模型看到的是一整条序列，而不是 300 个互不相关的点。
- 这类思路和前面的回归、聚类分册有本质差别。

## 5. 三大问题的直观角色分工

### 参数速览（本节）

适用问题（本节）：

1. 评估
2. 解码
3. 学习

| 问题 | 直观问题 |
|---|---|
| 评估 | 这个模型生成当前观测序列的概率有多大？ |
| 解码 | 最可能的隐状态路径是什么？ |
| 学习 | 只给观测序列时，参数该怎么估计？ |

### 理解重点

- 当前流水线最直接展示的是“学习 + 解码”两部分。
- 训练阶段通过 `fit(...)` 学习模型参数，预测阶段通过 `predict(...)` 输出隐状态序列。
- 文档后续章节就会围绕这两层工程实现展开。

## 6. 直觉如何映射到当前训练日志和输出

### 参数速览（本节）

适用输出项（分项）：

1. `n_components`
2. `n_iter`
3. `tol`
4. `state_pred`
5. `transmat_`

| 输出项 | 可以帮助判断什么 |
|---|---|
| `n_components` | 当前假设有多少个隐状态 |
| `n_iter`、`tol` | 训练迭代控制方式 |
| `state_pred` | 模型解码得到的隐状态路径 |
| `transmat_` | 学到的状态转移结构 |

### 理解重点

- 当前训练日志最重要的结构信息，不是系数或聚类中心，而是隐状态数和训练迭代设定。
- 当前流水线评估最直观的结果，是预测隐状态和真实隐状态的对比准确率，以及学习到的转移矩阵。
- 这说明 HMM 的结果解释重点在“状态路径”和“状态转移”，而不是点云几何结构。

## 常见坑

1. 把 `state_true` 当成监督标签，误以为当前 HMM 是监督学习。
2. 只盯着单个观测符号，不把它放回整条序列中理解。
3. 把 HMM 当成普通分类器，忽略它最重要的是建模“隐藏状态随时间演化”。

## 小结

- HMM 的核心直觉，是用一条看不见的状态链去解释看得见的观测序列。
- 转移矩阵负责描述状态如何演化，发射矩阵负责描述状态如何产生观测。
- 当前手工构造的离散序列数据，正是为了把这种“隐状态驱动观测”的结构尽量直观地展示出来。
